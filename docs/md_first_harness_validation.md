# MD-First Harness Validation Boundary

## Goal

Validate whether a China-specific forecast run is controlled by the Markdown protocol and the local main agent, not by an external code pipeline.

## Required Input Boundary

The run starts from:

```text
Q
resolution rule
optional p_m / market metadata
```

The user should then wait for:

```text
forecast_report.md
forecast_report.json
forecast_report.pdf
```

## High-Level Intuition Checks

1. Markdown defines the workflow.
   - Check `task.md` states that no code controller owns the loop.
   - Check `harness_protocol.md` is copied into the workspace.

2. The agent owns the run.
   - Check `plan.md` is written as the agent's plan, not as a fixed script.
   - Check `trajectory.md` contains agent decisions, not only tool logs.

3. Tools are capabilities.
   - Check tool calls appear only after an information gap is stated.
   - Check the same run does not blindly call a fixed sequence when evidence says otherwise.

4. Search is evidence-driven.
   - Check each evidence-producing action is followed by `agent_reviews/*.md`.
   - Check each review contains `observation`, `assessment`, `information_gap`, and `next_search_decision`.
   - Check the next search follows from the previous gap.

5. Polymarket is a baseline.
   - Check `p_m` is used as baseline/context only.
   - Check Polymarket pages are not used as factual evidence.

6. Report is agent output.
   - Check `forecast_report.md/json` is generated as the terminal artifact of the run.
   - Check report evidence paths trace back to `sources/`, `search_actions/`, or `agent_reviews/`.
   - Check `forecast_report.pdf` is rendered from the report files.

## Pass Criteria

A run passes only if artifacts show:

```text
Q + resolution -> task.md -> agent-owned plan -> evidence search -> review -> next search or stop -> report
```

The run fails if the report is assembled outside the agent loop or if `trajectory.md` is mostly a tool-call list.
